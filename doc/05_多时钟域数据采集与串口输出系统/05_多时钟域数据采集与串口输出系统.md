## 二、系统架构

```
                    ┌──────────────────────────────────────────────────┐
                    │            sampler_uart_top                      │
                    │                                                  │
 clk_fast(200MHz)──►│  ┌──────────────┐    ┌────────────┐             │
                    │  │ data_sampler  │───►│ async_fifo  │            │
 rst_n ────────────►│  │  (200MHz域)   │    │ (跨时钟域)  │            │
                    │  └──────────────┘    └─────┬──────┘            │
                    │                            │                    │
  clk_sys(50MHz)───►│                      ┌─────▼──────┐            │
                    │                      │  uart_tx    │───► tx_out │
                    │                      │  (50MHz域)  │            │
                    │                      └────────────┘            │
                    └──────────────────────────────────────────────────┘
```

数据流：`data_sampler` 在 200 MHz 时钟域产生采样数据，写入异步 FIFO；`uart_tx` 在 50 MHz 时钟域从 FIFO 读取数据，按 UART 协议串行发送。

## 三、模块设计

### 3.1 sampler_uart_top — 顶层模块

将三个子模块连接，并对外暴露时钟、复位与 UART 输出。

**参数：**

| 参数名       | 默认值      | 说明                     |
| ------------ | ----------- | ------------------------ |
| DATA_WIDTH   | 8           | 数据位宽                 |
| BAUD_RATE    | 115200      | UART 波特率              |
| CLK_SYS_FREQ | 50_000_000 | 系统时钟频率（Hz）       |

**端口：**

| 信号名   | 方向   | 位宽 | 说明                           |
| -------- | ------ | ---- | ------------------------------ |
| clk_fast | input  | 1    | 快时钟，200 MHz                |
| clk_sys  | input  | 1    | 系统时钟，50 MHz               |
| rst_n    | input  | 1    | 全局异步复位，低有效           |
| tx_out   | output | 1    | UART 串行数据输出              |
| fifo_full| output | 1    | FIFO 满指示（调试用）          |
| fifo_empty| output| 1    | FIFO 空指示（调试用）          |

### 3.2 data_sampler — 数据采集模块

工作在 200 MHz 时钟域。内部维护一个递增计数器作为模拟采样数据源。当 FIFO 未满时，以可控速率将数据写入 FIFO。

**参数：**

| 参数名      | 默认值 | 说明                               |
| ----------- | ------ | ---------------------------------- |
| DATA_WIDTH  | 8      | 输出数据位宽                       |
| SAMPLE_DIV  | 20000  | 采样分频系数（每 N 个时钟产生一笔） |

**端口：**

| 信号名     | 方向   | 位宽       | 说明                         |
| ---------- | ------ | ---------- | ---------------------------- |
| clk        | input  | 1          | 200 MHz 时钟                 |
| rst_n      | input  | 1          | 异步复位，低有效             |
| data_out   | output | DATA_WIDTH | 采样数据输出                 |
| data_valid | output | 1          | 数据有效指示                 |
| fifo_full  | input  | 1          | FIFO 满信号（反压）          |

**功能描述：**
- 内部分频计数器每 `SAMPLE_DIV` 个时钟周期产生一次数据
- 数据为 8-bit 递增计数器值（0x00 ~ 0xFF 循环），串口输出形如 `00 01 02 ... FE FF 00 01 ...`，接收端通过观察数值是否连续递增即可判断数据完整性
- 当 `fifo_full` 为高时暂停产生数据（反压机制）
- `data_valid` 仅在产生有效数据且 FIFO 未满时拉高一个时钟周期

**`SAMPLE_DIV` 取值说明：**

UART 115200 baud、8N1 格式下，每字节需 10 bit，理论吞吐量为 115200 / 10 = 11520 字节/秒，即每字节约 86.8 μs。在 200 MHz 时钟下对应约 17361 个时钟周期。`SAMPLE_DIV` 取 20000（对应每笔间隔 100 μs，即 10000 字节/秒）略低于 UART 吞吐上限，确保 FIFO 不会被填满，串口输出连续无断裂。

### 3.3 异步 FIFO — Xilinx FIFO Generator IP

跨时钟域 FIFO，写端工作在 `clk_fast`（200 MHz），读端工作在 `clk_sys`（50 MHz）。不手写 RTL，使用 SynthPilot MCP 工具配置一个 Xilinx FIFO Generator IP，配置要求如下：

| 配置项       | 值           | 说明                           |
| ------------ | ------------ | ------------------------------ |
| IP 实例名    | async_fifo   |                                |
| 数据位宽     | 8            | 与 data_sampler 输出一致       |
| 深度         | 16           | 教程演示用，无需太大           |
| 异步模式     | 是           | 写时钟 200 MHz，读时钟 50 MHz  |
| 读模式       | First Word Fall Through | 便于与下游握手逻辑对接 |
| 存储类型     | Block RAM    | 默认即可                       |

IP 生成后，其典型端口为 `wr_clk`、`rd_clk`、`din`、`wr_en`、`full`、`dout`、`rd_en`、`empty` 等，顶层例化时按实际生成的端口连接。

### 3.4 uart_tx — UART 发送模块

工作在 50 MHz 时钟域。从 FIFO 读取数据，按标准 UART 协议（8N1）串行发送。

**参数：**

| 参数名    | 默认值      | 说明               |
| --------- | ----------- | ------------------ |
| CLK_FREQ  | 50_000_000  | 系统时钟频率（Hz） |
| BAUD_RATE | 115200      | 波特率             |

**端口：**

| 信号名     | 方向   | 位宽 | 说明                           |
| ---------- | ------ | ---- | ------------------------------ |
| clk        | input  | 1    | 50 MHz 时钟                    |
| rst_n      | input  | 1    | 异步复位，低有效               |
| tx_data    | input  | 8    | 待发送数据                     |
| tx_valid   | input  | 1    | 数据有效（握手信号）           |
| tx_ready   | output | 1    | 发送器就绪（握手信号）         |
| tx_out     | output | 1    | UART 串行输出（空闲态为高）    |

**功能描述：**
- 空闲时 `tx_out` 保持高电平，`tx_ready` 为高
- 当 `tx_valid` 与 `tx_ready` 同时为高时，锁存数据并开始发送
- 发送帧格式：1 起始位（低）+ 8 数据位（LSB first）+ 1 停止位（高）
- 内部波特率分频计数器从系统时钟分频产生 bit 时钟
- 发送过程中 `tx_ready` 拉低，发送完成后恢复

## 四、顶层连接逻辑

顶层模块 `sampler_uart_top` 的内部连接关系：

```
data_sampler.data_out      ──► async_fifo.din
data_sampler.data_valid    ──► async_fifo.wr_en（受 wr_rst_busy 门控）
async_fifo.full | wr_rst_busy ──► data_sampler.fifo_full

async_fifo.dout            ──► uart_tx.tx_data
~async_fifo.empty & ~rd_rst_busy ──► uart_tx.tx_valid
uart_tx.tx_ready & ~empty & ~rd_rst_busy ──► async_fifo.rd_en
```

读端握手逻辑：FIFO 非空时将数据送入 UART 发送器，需正确处理 FIFO 读使能与 UART 握手的配合，避免对空 FIFO 发起读操作。注意 `wr_rst_busy` / `rd_rst_busy` 期间禁止读写。

## 六、仿真验证 Checklist

Testbench 例化完整的 `sampler_uart_top`（内含 `data_sampler`、`async_fifo`、`uart_tx`），并在外部编写 UART 接收模型（`uart_rx_model`）对串行输出 `tx_out` 进行采样解码。对以下场景逐项验证并自动判定 PASS/FAIL：

### 6.1 测试用例

| 编号   | 测试场景                           | 通过条件                                                                 |
| ------ | ---------------------------------- | ------------------------------------------------------------------------ |
| TC-01  | 复位初始状态                       | 复位释放后，`tx_out` 为高（空闲态），`fifo_empty` 为高，`fifo_full` 为低 |
| TC-02  | 单字节 UART 发送帧格式            | UART 接收模型解码出完整帧：1 起始位（低）+ 8 数据位（LSB first）+ 1 停止位（高） |
| TC-03  | 数据完整性（连续递增验证）         | UART 接收模型连续接收至少 16 字节，数值为 0x00→0x01→…→0x0F 连续递增     |
| TC-04  | 跨时钟域数据一致性                 | 200 MHz 域写入 FIFO 的数据与 50 MHz 域 UART 发出的数据逐字节比对，无丢失无错位 |
| TC-05  | FIFO 反压机制                      | 将 `SAMPLE_DIV` 设为极小值（如 2）使写入速率远超读取速率，验证 `fifo_full` 拉高后 `data_sampler` 暂停写入，FIFO 不溢出 |
| TC-06  | FIFO 空状态处理                    | 将 `SAMPLE_DIV` 设为极大值使 FIFO 长期为空，验证 `uart_tx` 不发送虚假数据，`tx_out` 保持空闲高电平 |
| TC-07  | 波特率精度                         | 测量 UART 输出每个 bit 的持续时间，与理论值（1/115200 s ≈ 8.68 μs）偏差不超过 ±2% |
| TC-08  | 长时间连续运行                     | 系统连续运行并成功接收至少 256 字节（覆盖计数器 0x00~0xFF 完整一轮），全部数据正确 |
| TC-09  | 复位恢复                           | 系统正常运行中途触发复位，释放后系统重新正常工作，UART 输出有效数据       |
| TC-10  | FIFO 满到非满恢复                  | 触发 FIFO 满后恢复正常速率，验证数据流恢复畅通，后续字节仍连续递增       |

### 6.2 UART 接收模型（uart_rx_model）

Testbench 内需实现一个行为级 UART 接收模型，用于自动解码 `tx_out` 信号：

- 检测起始位下降沿后，以波特率中点采样 8 个数据位
- 校验停止位为高电平
- 将每字节解码结果存入队列或数组，供各测试用例比对

### 6.3 Testbench 参数化

为便于不同测试场景复用同一 testbench，建议通过参数覆盖调整行为：

| 参数名               | 默认值   | 用途                                       |
| -------------------- | -------- | ------------------------------------------ |
| TB_SAMPLE_DIV        | 200      | 覆盖 `data_sampler` 的 `SAMPLE_DIV`（加速仿真） |
| TB_BAUD_RATE         | 1000000  | 覆盖波特率（1 Mbaud 加速仿真）             |
| TB_NUM_BYTES         | 256      | TC-08 长时间运行接收的目标字节数           |
| SIM_TIMEOUT_US       | 50000    | 仿真超时保护（μs），超时后强制结束并报 FAIL |

### 6.4 Testbench 输出要求

- 每个测试用例打印 `[PASS] TC-xx: 描述` 或 `[FAIL] TC-xx: 描述 — 期望值=xx, 实际值=xx`
- 全部完成后打印汇总：`=== x/10 PASSED ===`
- 使用 `$dumpfile` / `$dumpvars` 生成波形文件，以便 Vivado 查看
- UART 接收模型每成功解码一个字节打印 `[UART_RX] @%0t: 0x%02X`，便于调试

### 6.5 仿真加速建议

由于 UART 115200 波特率下每字节耗时约 86.8 μs，256 字节需 ~22 ms 仿真时间，在 200 MHz 时钟下约 440 万周期。为加速仿真：
- 可在仿真时将 `BAUD_RATE` 提高（如 1_000_000）并同步调整 `SAMPLE_DIV`
- 或减少 `TB_NUM_BYTES` 做快速回归，仅在最终验证时使用完整参数

## 八、综合后分析

综合完成后，打开综合后设计（`open_synthesized_design`），使用 SynthPilot MCP 工具进行分析。

> **关于分析阶段的说明：** 综合后可获得准确的资源利用率和网表级结构信息，时序分析基于估算线延迟（非实际布线），结果可用于早期问题发现但不作为签收依据。拥塞分析、布线状态、精确功耗估算需布局布线后才可获得，本阶段不涉及。

### 8.1 资源利用率分析 — 综合后可用，数据准确

- `report_utilization`：查看 LUT、FF、BRAM、DSP、IO 等资源使用量
- `report_utilization_hierarchical`：按模块层次分解，定位各子模块资源开销
- 确认总资源占用在 XC7Z010 容量范围内

### 8.2 时序分析 — 综合后可用，基于估算线延迟

- `report_timing_summary`：WNS / TNS / WHS / THS 汇总，确认各时钟域是否满足约束
- `analyze_critical_paths`：查看最差路径的逻辑级数与延迟分解
- `analyze_clock_domain_timing`：时钟对之间的 WNS 矩阵，关注跨时钟域路径
- `report_high_fanout_nets`：高扇出网络，识别潜在布线瓶颈
- **注意：** 综合后时序使用估算线延迟，实际 slack 在布局布线后会变化；此阶段主要用于发现大幅违例

### 8.3 时钟网络分析 — 综合后可用，数据准确

- `report_clock_networks`：确认 Clocking Wizard 输出的时钟树结构

### 8.4 CDC 分析 — 综合后可用

- `report_cdc`：跨时钟域路径报告
- `report_cdc_details`：详细的同步器识别与 CDC 路径分类
- 确认异步 FIFO 内部的 CDC 路径已被正确处理，无遗漏的裸跨时钟域信号

### 8.5 设计规则与方法学检查 — 综合后可用（部分规则）

- `report_drc`：设计规则检查，关注时钟、复位、I/O 相关告警

- **注意：** 部分 DRC 规则（如布线相关）仅在布局布线后生效，综合后检查覆盖的是子集

## 九、布局布线（Implementation）

### 9.1 实现策略

Vivado 提供多种实现策略，影响布局布线的优化目标和运行时间：

| 策略                          | 适用场景                                   |
| ----------------------------- | ------------------------------------------ |
| Vivado Implementation Defaults | 通用默认策略，平衡性能与编译时间           |
| Performance_Explore           | 时序紧张时使用，尝试更多优化但编译更慢     |
| Performance_ExtraTimingOpt    | 在 route 后追加时序优化迭代                |
| Area_Explore                  | 资源紧张时使用，优先面积优化               |
| Power_DefaultOpt              | 低功耗优先                                 |
| Flow_RuntimeOptimized         | 编译速度优先，牺牲部分优化质量             |

本设计综合后各时钟域时序裕量充足（200 MHz 域 WNS ≈ 1.3 ns，50 MHz 域 WNS ≈ 16 ns），资源利用率极低（< 1%），使用默认策略即可。

> **何时需要切换策略：** 当综合后 WNS 接近 0 或为负值时，应考虑 `Performance_Explore`；当资源利用率超过 70% 时，可能需要 `Area_Explore`。策略选择是**实现阶段最重要的旋钮之一**，但它不能修复 RTL 层面的架构问题——如果综合后时序违例严重（WNS < -2 ns），应优先优化 RTL 而非依赖工具策略。

### 9.2 运行布局布线

使用 SynthPilot MCP 工具 `run_implementation` 运行布局布线，流程自动经过三个阶段：

1. **opt_design** — 网表优化：常量传播、冗余逻辑删除、BRAM/DSP 推断优化
2. **place_design** — 布局：将逻辑单元映射到 FPGA 物理位置
3. **route_design** — 布线：为所有网络分配物理布线资源

完成后打开实现后设计（`open_implemented_design`）进入分析阶段。

## 十、实现后分析

实现后分析是设计签收的关键阶段。与综合后分析（第八节）不同，此时时序基于**真实布线延迟**，资源经过 opt_design 优化，且可获得拥塞、功耗等综合后无法获取的信息。

> **核心教学目标：** 通过综合→实现的逐项对比，理解"估算线延迟"与"真实布线延迟"的差异，建立对综合后时序报告可信度的直觉。

### 10.1 资源利用率

- `report_utilization`：查看实现后的 LUT、FF、BRAM、DSP、IO 资源使用量
- `report_utilization_hierarchical`：按模块层次分解

**与综合对比要点：**
- opt_design 可能优化掉部分冗余逻辑，导致 LUT/FF 数量减少
- BRAM、DSP、IO、时钟资源通常不变
- 若实现后资源明显增加，可能是工具插入了额外的布线辅助逻辑（如 LUT 用作路由）

### 10.2 时序分析（核心对比项）

- `report_timing_summary`：WNS / TNS / WHS / THS 汇总
- `analyze_critical_paths`：最差路径详细分解
- `analyze_clock_domain_timing`：时钟域对 WNS 矩阵
- `report_high_fanout_nets`：高扇出网络

**与综合对比要点（逐项对比表）：**

应生成如下对比表，每个时钟域分别对比：

| 指标                     | 综合后（估算） | 实现后（真实） | 变化   |
| ------------------------ | -------------- | -------------- | ------ |
| 200 MHz 域 Setup WNS    | x.xxx ns       | x.xxx ns       | Δ ns   |
| 200 MHz 域 Hold WHS     | x.xxx ns       | x.xxx ns       | Δ ns   |
| 50 MHz 域 Setup WNS     | x.xxx ns       | x.xxx ns       | Δ ns   |
| 50 MHz 域 Hold WHS      | x.xxx ns       | x.xxx ns       | Δ ns   |
| 跨域 200M→50M WNS       | x.xxx ns       | x.xxx ns       | Δ ns   |
| 跨域 50M→200M WNS       | x.xxx ns       | x.xxx ns       | Δ ns   |

**最差路径延迟分解对比：**

对同一条关键路径，对比综合后与实现后的延迟构成：

| 延迟成分      | 综合后（估算） | 实现后（真实） | 说明                                 |
| ------------- | -------------- | -------------- | ------------------------------------ |
| Logic Delay   | x.xxx ns       | x.xxx ns       | 由 LUT/FF 固有延迟决定，通常不变     |
| Route Delay   | x.xxx ns       | x.xxx ns       | 综合后为估算值，实现后为实际布线延迟 |
| Total         | x.xxx ns       | x.xxx ns       |                                      |
| Logic 占比    | xx%            | xx%            | 实现后 route 占比通常上升            |

> **教学重点：** Logic delay 在综合后和实现后几乎一致（由器件 speed file 决定），而 route delay 会发生显著变化。综合后的 route delay 是基于扇出和估算模型的预测值，实现后是基于实际物理位置的真实延迟。这解释了为什么综合后 WNS > 0 不等于时序一定满足——需要实现后确认。

### 10.3 布线质量分析

以下报告在综合后不可用，是实现阶段的独有分析能力：

- `report_route_status`：确认所有网络已成功布通，无 unrouted nets
- `report_congestion`：布线拥塞分析，识别拥塞热点区域。拥塞等级从 0（无拥塞）到 5+（严重拥塞），等级 ≥ 3 需要关注
- `report_design_analysis`：设计复杂度与布线质量综合评估

**关注要点：**
- 全部网络应为 fully routed 状态
- 拥塞等级应较低（本设计资源利用率 < 1%，预期无拥塞）
- 若出现拥塞热点，通常与高扇出网络、密集逻辑聚集相关

### 10.4 功耗估算

- `report_power`：基于实现后布线信息和默认活动率的功耗估算

**功耗分解维度：**
- 静态功耗（Leakage）：器件固有
- 动态功耗（Dynamic）：与时钟频率、切换活动率相关
- 按层次分解：Clocking / Logic / Signals / BRAM / IO

> **注意：** 默认活动率（通常 12.5%）是估算值。如需更准确的功耗分析，可通过仿真生成 SAIF 文件导入实际切换活动率，但本教程不涉及。

### 10.5 时钟网络（实现后更新）

- `report_clock_networks`：确认时钟树的物理实现
- `report_clock_utilization`：时钟资源实际放置位置（Clock Region 分布）
- `report_clock_interaction`：时钟域交互矩阵，确认 CDC 约束在实现后仍正确生效

### 10.6 DRC / 方法学检查

- `report_drc`：完整设计规则检查（含布线相关 DRC 规则）
- `report_methodology`：方法学检查

**与综合对比要点：**
- 综合后 DRC 只能覆盖部分规则（网表级），实现后可运行完整规则集（含布线级）
- 新增的布线相关 DRC 包括：布线冲突、天线效应、信号完整性等
- 对比综合后已知的告警是否仍然存在，是否有新增告警

### 10.7 CDC 复查

- `report_cdc`：跨时钟域路径报告
- `report_cdc_details`：同步器详细信息

确认实现后所有 CDC 路径仍标记为 Safe，实际布线未引入新的跨时钟域风险。

### 10.8 实现分析报告

分析完成后，在 `F:\synthpilot_tutorial\doc` 目录下生成实现后分析报告，内容应包含：

1. 资源利用率汇总（含综合→实现对比，标注 opt_design 优化效果）
2. **时序分析对比表**（综合估算 vs 实现真实，逐时钟域对比 WNS/WHS）
3. **最差路径延迟分解对比**（logic delay vs route delay 变化分析）
4. 布线状态确认（全部网络 fully routed）
5. 拥塞分析结果
6. 功耗估算（静态 + 动态，按层次分解）
7. DRC / 方法学检查结果（含与综合后对比，标注新增规则）
8. CDC 复查结果
9. **综合→实现变化总结**：归纳关键指标变化，给出设计签收结论

## 十一、比特流生成与下板验证

### 11.1 生成比特流

实现完成且时序签收通过后，使用 SynthPilot MCP 工具 `generate_bitstream` 生成比特流文件（.bit）。

### 11.2 下载到开发板

使用 SynthPilot MCP 工具 `program_device` 将比特流下载到 FPGA。下载前确保开发板已通过 JTAG/USB 连接并被 Vivado Hardware Manager 识别。

> **关于 ZPS7-1 告警：** 本设计未例化 PS7 模块。在纯 PL 模式下，部分 Zynq 开发板可能需要额外操作（如短接 MIO 引脚或使用 JTAG Boot Mode）才能正常加载。若下载后功能异常，优先排查此问题。

### 11.3 使用 SynthPilot 串口工具初步验证

在编写上位机之前，先使用 SynthPilot MCP 提供的串口工具进行快速验证：

1. **打开串口**：配置串口参数——波特率 115200，数据位 8，无校验，停止位 1（8N1），端口号根据实际连接选择
2. **读取数据**：接收若干字节（如 32~64 字节），以十六进制格式显示
3. **人工检查**：确认数据为连续递增序列（`00 01 02 03 ... FE FF 00 01 ...`），无跳变、无重复

此步骤的目的是快速确认硬件链路畅通、FPGA 功能正常，再进入上位机开发。

## 十二、上位机开发

### 12.1 技术栈

### 12.2 功能需求

**核心功能：**

1. **串口接收**：以 115200 baud 8N1 持续接收数据
2. **实时波形显示**：将接收到的字节值（0~255）绘制为滚动波形图，呈现锯齿波形态
3. **数据完整性校验**：自动检测相邻字节是否满足递增关系（允许 255→0 回绕），统计错误次数

**界面布局：**

```
┌─────────────────────────────────────────────────────┐
│  串口数据可视化工具                          [串口配置] │
├─────────────────────────────────────────────────────┤
│                                                     │
│              实时滚动波形图                           │
│        Y轴: 0~255 (字节值)                           │
│        X轴: 采样点序号 (滚动窗口)                     │
│                                                     │
├─────────────────────────────────────────────────────┤
│  已接收: xxxx 字节 │ 错误: x 次 │ 当前值: 0xNN       │
│  [打开串口]  [关闭串口]                              │
└─────────────────────────────────────────────────────┘
```

**界面要素：**

| 元素         | 说明                                              |
| ------------ | ------------------------------------------------- |
| 波形区域     | pyqtgraph PlotWidget，显示最近 1000 个数据点的滚动曲线 |
| 串口配置     | 端口号下拉框（自动枚举）、波特率（默认 115200）    |
| 状态栏       | 已接收字节数、错误计数、当前字节值                 |
| 控制按钮     | 打开/关闭串口                                     |

### 12.3 关键设计要点

**串口读取线程：**
- 使用 `QThread` 或 `QTimer` 避免阻塞 GUI
- 推荐方案：`QThread` + 信号槽，在工作线程中阻塞读取，通过信号将数据传递到主线程更新 UI

**数据完整性校验逻辑：**
```
若 current_byte != (previous_byte + 1) % 256:
    error_count += 1
```

**波形更新策略：**
- 维护一个固定长度（如 1000 点）的环形缓冲区
- 每收到新数据追加到缓冲区，超出部分自动丢弃最早的数据
- 使用 pyqtgraph 的 `setData()` 方法批量更新，避免逐点更新造成的性能问题
