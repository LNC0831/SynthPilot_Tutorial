## 2. 时序分析（核心对比项）

### 2.1 总体时序汇总

| 指标  | 综合后（估算） | 实现后（真实） | 变化    | 失败端点数 | 总端点数 | 结论 |

| WNS   |      1.319 ns  |      0.952 ns  | -0.367  |          0 |      257 | MET  |
| TNS   |      0.000 ns  |      0.000 ns  |  0.000  |          0 |      257 | MET  |
| WHS   |      0.045 ns  |      0.101 ns  | +0.056  |          0 |      249 | MET  |
| THS   |      0.000 ns  |      0.000 ns  |  0.000  |          0 |      249 | MET  |
| WPWS  |      1.520 ns  |      1.520 ns  |  0.000  |          0 |      173 | MET  |

**所有用户指定的时序约束均满足。**

### 2.2 各时钟域时序对比表

| 指标                     | 综合后（估算） | 实现后（真实） | 变化 (Δ ns) |
| ------------------------ | -------------: | -------------: | ----------: |
| 200 MHz 域 Setup WNS     |      1.319 ns  |      0.952 ns  |      -0.367 |
| 200 MHz 域 Hold WHS      |      0.045 ns  |      0.101 ns  |      +0.056 |
| 50 MHz 域 Setup WNS      |     16.149 ns  |     15.623 ns  |      -0.526 |
| 50 MHz 域 Hold WHS       |      0.045 ns  |      0.122 ns  |      +0.077 |
| 跨域 200M→50M WNS        |      3.990 ns  |      3.666 ns  |      -0.324 |
| 跨域 50M→200M WNS        |     18.990 ns  |     18.580 ns  |      -0.410 |

**分析：**

- **Setup（建立时间）**：实现后 WNS 普遍下降约 0.3~0.5 ns，符合预期——真实布线延迟大于综合估算线延迟
- **Hold（保持时间）**：实现后 WHS 普遍改善（数值增大），这是因为实现工具在布局布线时主动插入延迟以满足保持时间约束
- 200 MHz 域 WNS = 0.952 ns，裕量仍然健康（占周期 19%）
- 50 MHz 域 WNS = 15.623 ns，裕量极大（占周期 78%）

### 2.3 关键路径分析（Top 5）

| 排名 | Slack (ns) | 逻辑级数 | 源→目的                              | 时钟域     |
| ---: | ---------: | -------: | ------------------------------------ | ---------- |
|    1 |      0.952 |        2 | data_valid_reg → BRAM WEBWE[3]       | 200 MHz 域 |
|    2 |      1.048 |        2 | data_valid_reg → BRAM WEBWE[2]       | 200 MHz 域 |
|    3 |      1.104 |        3 | div_cnt_reg[10] → data_valid_reg     | 200 MHz 域 |
|    4 |      1.104 |        2 | data_valid_reg → BRAM WEBWE[1]       | 200 MHz 域 |
|    5 |      1.201 |        2 | data_valid_reg → BRAM WEBWE[0]       | 200 MHz 域 |

**关键路径变化说明：**

综合后最差路径为 `div_cnt` 进位链（4 级逻辑），实现后最差路径变为 `data_valid_reg → BRAM WEBWE`（2 级逻辑）。这说明 opt_design 优化了进位链路径的布局，但 `data_valid_reg` 到 BRAM 写使能的布线距离较长，成为新的瓶颈。

### 2.4 最差路径延迟分解对比

| 延迟成分      | 综合后（估算） | 实现后（真实） | 说明                                             |
| ------------- | -------------: | -------------: | ------------------------------------------------ |
| Logic Delay   |   1.877 ns (53.8%) |   0.698 ns (22.2%) | 实现后关键路径逻辑级数从 4 级降至 2 级      |
| Route Delay   |   1.609 ns (46.2%) |   2.441 ns (77.8%) | 实现后布线延迟显著增加，成为延迟主导         |
| Total         |      3.486 ns  |      3.139 ns  |                                                  |

> **教学重点：** 综合后与实现后的关键路径并非同一条路径，因此不宜直接对比单条路径的 logic/route 比值。但总体规律成立：**实现后 route delay 占比显著上升**（从 46.2% 到 77.8%），这是因为综合后的 route delay 是基于扇出和估算模型的预测值，而实现后是基于实际物理位置的真实延迟。对于本设计，即使 route 占比大幅上升，时序仍然满足，说明 200 MHz 对 XC7Z010-1 speed grade 而言是合理的目标频率。

## 5. 时钟网络（实现后）

### 5.1 时钟树结构

```
sys_clk (50 MHz, K17 引脚, IOB_X0Y76)
  └── IBUF → MMCME2_ADV (X1Y1 Clock Region)
        ├── CLKOUT0 → BUFG (X0Y17) → clk_out1 (200 MHz, 79 loads, Clock Region X1Y0)
        ├── CLKOUT1 → BUFG (X0Y16) → clk_out2 (50 MHz, 86 loads, Clock Region X1Y0)
        └── CLKFBOUT → BUFG (X0Y18) → clkfbout (反馈时钟, 1 load, Clock Region X1Y1)
```

### 5.2 时钟资源使用

| 资源类型 | 使用量 | 可用量 |
| -------- | -----: | -----: |
| BUFGCTRL |      3 |     32 |
| MMCM     |      1 |      2 |
| PLL      |      0 |      2 |

与综合后一致，时钟资源无变化。

### 5.3 时钟域交互矩阵

| 源时钟 → 目的时钟                   | WNS (ns) | 分类             | 约束状态                |
| ----------------------------------- | -------: | ---------------- | ----------------------- |
| clk_out1 → clk_out1 (200M intra)   |     0.95 | Clean            | Timed                   |
| clk_out2 → clk_out2 (50M intra)    |    15.62 | Clean            | Timed                   |
| clk_out1 → clk_out2 (200M→50M)     |     3.67 | Ignored          | Max Delay Datapath Only |

时钟域分类和约束状态与综合后一致，实现后 CDC 约束正确生效。

### 5.4 物理布局

所有功能逻辑和时钟负载集中在 **Clock Region X1Y0**，MMCM 位于 **Clock Region X1Y1**。布局紧凑，时钟路径短，有利于时序收敛。

## 6. DRC / 方法学检查

### 6.1 DRC 检查

| 规则 ID    | 严重级别 | 综合后 | 实现后 | 描述                              |
| ---------- | -------- | -----: | -----: | --------------------------------- |
| REQP-1840  | Warning  |      6 |      6 | RAMB18 async control check        |
| ZPS7-1     | Warning  |      1 |      1 | PS7 block required                |
| RTSTAT-10  | Warning  |    N/A |      1 | No routable loads（新增）         |

**新增规则说明：**

- **RTSTAT-10**（实现后新增）：异步 FIFO 内部有 9 条无可布线负载的网络。这是 FIFO IP 内部未使用的诊断/状态信号（如 `aempty_fwft_i`、`user_valid`、`ram_empty_i` 及复位寄存器），属于 IP 正常行为，不影响功能。

**已知告警处理：**

- **REQP-1840**（6 条）：与综合后相同，BRAM 异步控制引脚告警。FIFO IP 内部 `wr_rst_busy` / `rd_rst_busy` 机制确保安全，可安全忽略。
- **ZPS7-1**（1 条）：与综合后相同，纯 PL 教学场景下可接受。

### 8.2 设计签收结论

1. **时序签收：通过。** 所有时钟域 WNS > 0，TNS/THS 均为零。200 MHz 域 WNS = 0.952 ns，占周期 19%，裕量健康。综合→实现 WNS 降幅 0.367 ns，在可接受范围内。

2. **资源签收：通过。** 总资源利用率 < 1%，XC7Z010 容量极为充裕。opt_design 有效优化了冗余逻辑。

3. **布线签收：通过。** 全部网络 fully routed，零拥塞，零布线错误。

4. **功耗签收：通过。** 总功耗 0.203 W，结温 27.3°C，远低于器件额定值。

5. **CDC 签收：通过。** 所有跨时钟域路径标记为 Safe，实现后未引入新风险。

6. **DRC 签收：通过（有条件）。** 8 条 Warning 均为已知的可接受告警（BRAM 异步控制、PS7 未例化、IP 内部无负载网络），无 Error。

**综合评定：设计已通过实现后签收，可进入比特流生成阶段。**
